1. Modules as compile-time values (signatures, parameterized modules) — literally doesn't exist yet: no signature keyword, no grammar, no symbol-table concept. Greenfield work, but self-contained (doesn't block other features).
2. ~~Dependent/refinement types~~ — **done**. Implemented per `spec/dependent-types-implementation-plan.md` against the design in `spec/dependent-types-design.md`: symbolic const generics, declaration-nominal refinement types, the quantifier-free linear-integer Reasoning solver (`src/semantic/reason.*`), `try_from` typed runtime proofs, and flow-sensitive narrowing. What remains is listed under item 4 and item 5 below.
3. Higher-kinded traits — parses, zero semantic model of type constructors.
4. Contract enforcement, runtime half — contracts are now *reasoned* about (a statically refuted `pre` at a call site is a compile error, a statically proven one is recorded in `checked_types::elided_contracts` so lowering can omit it), but an *unproven* `pre`/`post`/`invariant` is still not lowered to a runtime assertion in bytecode/LLVM. The checker already tells lowering exactly which checks it may skip; what's missing is emitting the check-and-panic for the rest, plus honouring the release-elision flag.
5. Const-generic monomorphization — lowering receives fully-solved const values, and `try_from` on a still-generic refinement (`index[n].try_from(x)` inside a function generic over `n`) is diagnosed rather than compiled, because `n` has no runtime existence without monomorphizing the body per instantiation.
6. `if let` is not lowered to HIR at all (pre-existing, unrelated to refinements) — so the spec's `if let @some(i) = index[n].try_from(raw)` idiom type-checks but must be written as a `match` to reach a backend. Likewise the LLVM backend has no sum-type representation yet (codegen increment 6), so any `option`-returning code — `try_from` included — runs on the VM but not through `--compile`.
