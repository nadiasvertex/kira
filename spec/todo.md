1. Modules as compile-time values (signatures, parameterized modules) — literally doesn't exist yet: no signature keyword, no grammar, no symbol-table concept. Greenfield work, but self-contained (doesn't block other features).
2. ~~Dependent/refinement types~~ — **done**. Implemented per `spec/dependent-types-implementation-plan.md` against the design in `spec/dependent-types-design.md`: symbolic const generics, declaration-nominal refinement types, the quantifier-free linear-integer Reasoning solver (`src/semantic/reason.*`), `try_from` typed runtime proofs, and flow-sensitive narrowing. What remains is listed under item 4 and item 5 below.
3. Higher-kinded traits — parses, zero semantic model of type constructors.
4. Contracts on a `generator def` (its body runs in steps, so entry/exit don't mean what they mean for a call — lowering rejects them rather than checking at the wrong times).
5. ~~Const-generic monomorphization~~ — **done**. A free function generic only over compile-time *values* (`def get[n: usize](v: array[int32, n], i: index[n])`) is compiled once per constant its call sites pin `n` down to: `semantic::checker::instantiate_const_generic` (`src/semantic/check.cpp`) clones the declaration, re-checks it with `n` bound to that constant, and registers it as `get$3`; `hir::lower_module` lowers the instances and skips the template, which has no runtime form. So lowering receives fully-solved const values, `index[n].try_from(x)` compiles into a real check against a real bound, and a value use of `n` inside the body becomes the literal. Remaining gaps: a value parameter no argument determines (`def zeros[n: usize]() -> array[int32, n]`, with no explicit `f[8]()` call syntax) is diagnosed, not solved; const-generic *methods* (on an `impl`/`extend` target) still fall back to the template path and are refused by lowering; and *type*-generic monomorphization (item 3's neighbour — `def identity[T](x: T)`) remains unimplemented.
