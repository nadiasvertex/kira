7. Numeric conversion calls do not lower. `int32(xs.len())` and `int32(a)` for an `int64` `a` both type-check and then fail with "no concrete checked type is available for this node", pointing at the conversion callee. Since Kira never converts numerically implicitly, this leaves mixed-width arithmetic (notably anything combining `usize` from `.len()` with a sized integer) with no expressible form.
8. A generic function whose return type is a generic struct declared in a *third* module type-checks and then fails to lower. `def map[I, T, U](...) -> map_iter[I, T, U]` in `std.algo`, with `map_iter` declared in `std.iter` and brought in by `use std.iter.map_iter`, reaches lowering with "no concrete checked type is available for this node" at the call site. The same code works when the struct is declared in the same module as the function, and works cross-module when the function is *not* generic â€” it is the combination that fails, so the instantiated return type is not being resolved through the declaring file's member imports. `std.algo` sidesteps it by declaring each adapter type beside its constructor, which is the better arrangement anyway, but the gap will bite any user who splits a generic API across modules.
9. A qualified type path through an imported module's last segment does not resolve. After `use std.iter`, writing `iter.map_iter` reports ``qualified type path `iter.map_iter` does not resolve from module `std.algo` `` with the note "checked absolute path `std.algo.iter.map_iter`" â€” the path is interpreted relative to the current module rather than through the import binding. Member imports (`use std.iter.map_iter`) work, so there is a spelling that does the job, but the diagnostic sends the user toward declaring the symbol locally rather than toward the import that would fix it.


10. Most of the builtin method table type-checks and then does not lower. `k_builtin_methods` (`src/semantic/check.cpp`) is a hardcoded type-rule table, not a set of real declarations, so `resolved_callees_` never points at one and such calls cannot go through ordinary method-call lowering â€” each needs an explicit interception in `lower_call`. Only four exist (`len`, `as_bytes`, `push`, `generator.next`), so **14 of the table's 21 entries are broken**: `option` `unwrap`/`unwrap_or`/`is_some`/`is_none`, `result` `unwrap`/`is_ok`/`is_err`, `list` `pop`/`first`/`last`/`find`/`filter`/`contains`/`any`/`all`/`sort`/`clear`/`is_empty`, and `slice` `is_empty`. Each fails with "no concrete checked type is available for this node" pointing at the *callee* (`o.unwrap`), because lowering falls through to treating the method name as a struct field. The table's own doc comment claims "Only methods with real lowering belong here", which is false and should be fixed alongside. `match` is the working spelling for the `option`/`result` cases, which is why nothing in `std.iter`/`std.algo` calls `unwrap` internally, but these are among the first methods a user reaches for. Per the minimal-intrinsics philosophy the fix is to write these in Kira in the stdlib using `extend[T] option[T]:` etc. â€” now possible, since parameterized `extend` works â€” rather than adding a dozen more C++ interceptions to `lower_call`.

11. Tuples have no field access and no nested payload patterns. `t.0` is a parse error ("expected a field name after `.`"), and `@some((a, b))` type-checks but fails to lower with "variant `some`'s payload pattern needs a non-wildcard sub-pattern ... only plain bindings (`@some(x)`) are supported". Together these mean a tuple can only be taken apart by a `let (a, b) = t` in a statement position, so every consumer of `zip`/`enumerate` must bind the pair whole and destructure it on the next line. This is the R3a gap from the design, slightly wider than recorded there: the design notes `for (k, v)` fails, but the same limitation applies to any pattern nesting a tuple, and `.0` closes off the other obvious spelling. Every map iterator in the collections design yields tuples, so this sits on the critical path.
