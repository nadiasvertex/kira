1. Modules as compile-time values (signatures, parameterized modules) — literally doesn't exist yet: no signature keyword, no grammar, no symbol-table concept. Greenfield work, but self-contained (doesn't block other features).
2. ~~Dependent/refinement types~~ — **done**. Implemented per `spec/dependent-types-implementation-plan.md` against the design in `spec/dependent-types-design.md`: symbolic const generics, declaration-nominal refinement types, the quantifier-free linear-integer Reasoning solver (`src/semantic/reason.*`), `try_from` typed runtime proofs, and flow-sensitive narrowing. What remains is listed under item 4 and item 5 below.
3. Higher-kinded traits — parses, zero semantic model of type constructors.
4. ~~Contract enforcement, runtime half~~ — **done**. An unproven `pre`/`post`/`invariant` now lowers to a `hir_contract_check` that both backends compile to a check-and-panic (`panic_reason::precondition_violated`/`postcondition_violated`/`invariant_violated`): a `pre` at the callee's entry, a `post` at each exit (with `return` — newly a legal name in a postcondition — bound to the value being returned), an `invariant` at each construction and field write. `checked_types::elided_contracts` was redefined along the way: a call-site proof can't license omitting a check the callee makes on behalf of *every* caller, so a `pre` is now elided only when it follows from the parameters' own types (`collect_function_facts`), which no call can violate. `--no-contract-checks` is the spec's release elision. Not covered: contracts on a `generator def` (its body runs in steps, so entry/exit don't mean what they mean for a call — lowering rejects them rather than checking at the wrong times).
5. Const-generic monomorphization — lowering receives fully-solved const values, and `try_from` on a still-generic refinement (`index[n].try_from(x)` inside a function generic over `n`) is diagnosed rather than compiled, because `n` has no runtime existence without monomorphizing the body per instantiation.
6. ~~`if let` lowering~~ — **done**. The spec's `if let @some(i) = index[n].try_from(raw)` idiom now reaches both backends; it no longer has to be rewritten as a `match`. An `if let`/`elif let` branch has no boolean condition to test, so it lowers to a two-arm `hir_match` over its scrutinee (`lower_if_let_chain`, `src/hir/lower.cpp`): the pattern's arm carries the branch body, and a wildcard arm carries everything after that branch (the remaining `elif`s and the `else`, lowered recursively), so a chain mixing both branch forms becomes alternating `hir_if`/`hir_match` nodes. No new HIR node kind, no backend change. Two fixes came with it: the checker used to check an `if let` branch's pattern *before* opening the branch's scope, so its bindings leaked past the `if` (`while let` never did this); and `llvm_codegen` typed an `if` in expression position with the scalar-only `numeric_kind_for`, so an `if` yielding a heap value failed to compile — which is what actually blocked `try_from` under `--compile`, since it desugars to an `if`/`else` whose value is an `option`. The LLVM backend does have a sum-type representation now (an earlier `# 6` claim to the contrary was stale): `match` on an `option` already compiled.

   Not covered: `if let` in *expression* position (`let n = if let @some(x) = v: x else: 0`) does not parse — the parser's expression-form `if` accepts no `let` branch, though `ast::if_expr` has the fields for one. A tail `if let` in a block is value-producing and works, so this is a parser gap, not a lowering one.
