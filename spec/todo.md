8. A struct literal whose head carries explicit type arguments does not parse. `box[int32] { cur: @none }` reports "expected end of line after this" pointing at the `]`, so an explicitly-instantiated generic struct can only be built where inference supplies the arguments (a `return` into a declared type, an annotated `let`). The module-qualified head `q.holder { ... }` was the same shape of gap and is now fixed; this is the remaining one — `parse_postfix` does not consider a `{` after a bracketed type-argument list. Found while testing the generic-impl method fix.
9. A self-qualified module path does not resolve. With `module main` and a `module inner` submodule declared in the same file, `main.inner.holder` reports "does not resolve from module `main`" even though the note in that very diagnostic points at `module inner` as the nearest resolved module. Fails identically in type position and as a struct-literal head, so it is a module-graph gap rather than a parser one. The two-file spelling (`use p.q` then `q.holder`) works. Found while writing the qualified-struct-literal test.
10. The multi-line sum-type form does not parse. `spec/kira-reference.md` documents it in three places (lines 445, 512, 594) as a `=` followed by indented `| @variant` lines, and it reports "expected end of type block (dedent)" pointing at the first `|`. Only the single-line form (`type direction = @north | @south`) works, which is what every declaration in `src/std` uses — so the gap has stayed invisible. Either the parser accepts the documented form or the reference stops documenting it; the former, since the multi-line form is the readable one for a sum type with payloads.
11. `deriving ord` type-checks but `cmp` does not lower: `a.cmp(b)` on a `deriving ord` type fails with "no concrete checked type is available for this node". `ord` is deliberately excluded from `k_real_derive_traits` (`src/semantic/check.cpp`) and stays on the type-check-only `derived_method_result` path. Its return type `ordering` is now a real sum type, so the remaining work is a real derived `cmp` — a lexicographic field-by-field comparison — which also needs `ord` impls for the primitive field types.
12. A compound bound in a `where` clause cannot be cloned for monomorphization. `def sum[I, T](it: I) -> T where I: iterator[T], T: add + zero` reports "cannot clone this construct for monomorphization: this type-expression shape" pointing at `add + zero`; splitting it to a single bound compiles. `std.algo`'s `sum` and `product` are written `T: zero` and `T: one` for this reason rather than the `T: add + zero` / `T: mul + one` design §6.1 specifies.
13. A lambda capturing a mutable local collection does not lower. `var seen: list[int32] = []` followed by `xs.into_iter().for_each((x) => seen.push(x * 2))` type-checks and then fails with "no concrete checked type is available for this node" pointing at `seen.push`. Found while testing `for_each`, whose only observable effect is a capture.
14. `zero`/`one` have no impls for the 128-bit types (`int128`, `uint128`, `float128`), because a reachable module is compiled in full and the bytecode VM has no representation for them yet — an `impl zero for int128` in `std.traits` makes every program fail to compile. They arrive with codegen increment 6, alongside the rest of 128-bit support.
