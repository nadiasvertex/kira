A struct literal whose head carries explicit type arguments does not parse. `box[int32] { cur: @none }` reports "expected end of line after this" pointing at the `]`, so an explicitly-instantiated generic struct can only be built where inference supplies the arguments (a `return` into a declared type, an annotated `let`). 9. A self-qualified module path does not resolve. With `module main` and a `module inner` submodule declared in the same file, `main.inner.holder` reports "does not resolve from module `main`" even though the note in that very diagnostic points at `module inner` as the nearest resolved module. Fails identically in type position and as a struct-literal head, so it is a module-graph gap rather than a parser one. The two-file spelling (`use p.q` then `q.holder`) works. Found while writing the qualified-struct-literal test.
