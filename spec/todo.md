7. `extend` on an *applied* generic type compiles no instance. `extend gen[int32]:` type-checks and then dies in lowering with ``call to `gen::doubled` could not be resolved to a function in this compiled module`` — the same type-check-but-compile-nothing split that `check_impl_generic_static_call` fixed for `static def` on generic impls, except here it affects instance methods too. The generic spelling `extend[T] gen[T]:` does not parse at all (`expected `:` but found identifier`): `extend` accepts no type parameter list, so a parameterized inherent impl has *no* working form. Bare `extend gen:` and `impl[T] Trait for gen[T]` both work, so the workaround is to route inherent methods on parameterized types through a trait. Pre-existing on master, unrelated to §5.1 `collect`.
8. Numeric conversion calls do not lower. `int32(xs.len())` and `int32(a)` for an `int64` `a` both type-check and then fail with "no concrete checked type is available for this node", pointing at the conversion callee. Since Kira never converts numerically implicitly, this leaves mixed-width arithmetic (notably anything combining `usize` from `.len()` with a sized integer) with no expressible form.
9. A generic function whose return type is a generic struct declared in a *third* module type-checks and then fails to lower. `def map[I, T, U](...) -> map_iter[I, T, U]` in `std.algo`, with `map_iter` declared in `std.iter` and brought in by `use std.iter.map_iter`, reaches lowering with "no concrete checked type is available for this node" at the call site. The same code works when the struct is declared in the same module as the function, and works cross-module when the function is *not* generic — it is the combination that fails, so the instantiated return type is not being resolved through the declaring file's member imports. `std.algo` sidesteps it by declaring each adapter type beside its constructor, which is the better arrangement anyway, but the gap will bite any user who splits a generic API across modules.
10. A qualified type path through an imported module's last segment does not resolve. After `use std.iter`, writing `iter.map_iter` reports ``qualified type path `iter.map_iter` does not resolve from module `std.algo` `` with the note "checked absolute path `std.algo.iter.map_iter`" — the path is interpreted relative to the current module rather than through the import binding. Member imports (`use std.iter.map_iter`) work, so there is a spelling that does the job, but the diagnostic sends the user toward declaring the symbol locally rather than toward the import that would fix it.
