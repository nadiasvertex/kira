5. DONE. Lambdas containing string interpolation (or any call to a module-level function) failed to lower ("captured variable could not be found"): `hir::free_variables` reports every unbound referenced symbol, which includes module-level function callees (an ordinary call's callee is a `hir_local_ref`, and string interpolation desugars into calls to `std.fmt` helpers). Both backends now filter the free-variable list down to symbols that are actually locals of the enclosing function (`compile_lambda_value` in `src/bytecode_compiler/compile.cpp` and `src/llvm_codegen/codegen.cpp`), so globals are resolved directly against the function table instead of being captured into — and read back out of — the closure environment. Covered by `036_lambda_captures_local_not_global.kira` (std-free core, both tiers) and `cli_test`'s `test_run_lambda_body_string_interpolation_captures` (full std session).
6. Tail call optimization
7. DONE. A hand-rolled type implementing `std.iter.iterator[T]` is now usable directly in a `for` loop. The checker (`check_body_node`'s `for_stmt` case in `src/semantic/check.cpp`) resolves a `next(mut self) -> option[T]` method on the iterable (new `try_resolve_iterator` helper, which `element_type_of` also consults so the loop variable types as `T`) and records an `iterator_loop_dispatch` (new struct in `src/semantic/types.h`, stored in `checked_types::for_iterator_dispatches`) keyed by the `ast::for_stmt`. Lowering (`hir::lower_iterator_loop` in `src/hir/lower.cpp`, dispatched from `lower_for_stmt` ahead of the indexed-loop fallthrough) desugars `for x in it: ...` into the same shape as a hand-written `while let @some(x) = it.next(): ...` — the handle is bound once (`next(mut self)` mutates it through the pointer each pass) and the `while_let` subject is a real `hir_call` to the `Type::next`-mangled method rebuilt from the dispatch, so both backends compile it exactly like a written method call (no new HIR node or backend change). Resolution is duck-typed on the `next(mut self) -> option[T]` shape rather than on trait conformance, so a locally-defined `iterator[T]` trait works too. Covered by `037_user_iterator_for_loop.kira` (std-free core, both tiers), `cli_test`'s `test_run_for_loop_over_user_std_iterator` (real `std.iter` session, exit code 10), and `check_test`'s `accept_for_over_user_iterator`.
8. DONE. Single-element indexing of `str`/`slice[T]`/`slice_mut[T]` now has codegen in both backends. `infer_index` (`src/semantic/check.cpp`) types a scalar `str` index as `byte` (and requires an integer key); `slice`/`slice_mut` already typed to their element. `compile_index` in both `src/llvm_codegen/codegen.cpp` and `src/bytecode_compiler/compile.cpp` element-indexes the shared 2-slot `{ len; data }` view (`len` from slot 0, byte data pointer from slot 1, bounds check, then a load at `index * element_stride`), reusing the same tail the array/list paths use; `array[byte, N]` stays on the fixed-array path since it is the object itself, not a header view. Covered by `035_str_slice_element_index.kira` (both tiers) and `check_test`'s `accept_str_byte_index`/`report_str_index_non_integer`. Deferred sibling: still no in-Kira migration of `std.string`'s `rt_str_*` byte work (a follow-up now unblocked, not required).
